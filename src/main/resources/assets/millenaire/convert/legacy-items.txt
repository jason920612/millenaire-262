// Declarative legacy 1.12 (item-name, metadata) -> modern 26.2 item registry-id conversion table.
//
// This is the data behind MillConvert.legacyItemToStack(LegacyItem). It captures the 1.12 -> 26.2
// item FLATTENING: in 1.12 a single item id carried a metadata/damage value that selected a variant
// (dye;4 = blue dye, wool;15 = black wool, planks;2 = birch planks, ...). In 26.2 each variant became
// a distinct registry id (blue_dye, black_wool, birch_planks, ...) and item metadata is gone.
//
// Format:  <legacy_name>:<meta> = <modern_item_id>
//   - <legacy_name> is the 1.12 item/block REGISTRY NAME without the "minecraft:" prefix.
//   - <meta>        is the 1.12 metadata int the variant was selected by.
//   - <modern_item_id> is the 26.2 registry id (with namespace) the (name, meta) pair resolves to.
//
// 1.12 evidence: the right-hand side is exactly what Millénaire's hand-migrated 26.2 itemlist.txt names
// for the same good key, and the left-hand (name, meta) is the original 1.12 itemlist.txt entry for that
// same key (decompiled/todeploy/millenaire/itemlist.txt). Each line below is therefore the verbatim
// 1.12->26.2 mapping pre-M4 InvItem.loadInvItemList produced.
//
// Only (name, meta) pairs whose modern id DIFFERS from a plain registry lookup of <legacy_name> need an
// entry: a plain item (meta 0, name unchanged 1.12->26.2 like "minecraft:diamond") is resolved directly
// by name and is NOT listed here. Mill custom items that kept their meta as a live in-item variant in
// 26.2 (stone_deco / wood_deco / extended_mud_brick / stained_glass / sod / bannerpattern) are likewise
// NOT here: their meta is carried on the InvItem, the name resolves directly, and the variant is a
// blockstate/data discriminator -- not a distinct registry id.
